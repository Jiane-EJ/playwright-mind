# 第一段探索建模每日开发计划

## 基本信息

* 日期：2026-03-13
* 作者：Jiane
* 目标：以最小改动方式补齐第一段探索建模能力，并能稳定向第二段交付任务草稿

## 开发节奏说明

本排期按 7 个工作日设计，默认前提如下：

* 继续复用当前仓库的 `Playwright + Midscene + Stage2 + SQLite`
* 不引入前端页面
* 不做大范围目录重构
* 每天都有可见交付物

## 每日开发内容

| 天数 | 当日目标 | 当日开发内容 | 当日产出 | 当日验收点 |
|------|----------|--------------|----------|------------|
| Day 1 | 明确第一段输入输出边界 | 定义第一段请求文件格式、探索结果格式、草稿任务格式；补充方案文档和 README 引用 | 第一段方案文档、每日排期文档、README 同步说明 | 第一段边界清晰，不再与前端页面、调度系统混在一起 |
| Day 2 | 搭建第一段最小入口 | 新增 `src/stage1/types.ts`、`request-loader.ts`、`stage1-runner.ts` 骨架；新增 `tests/generated/stage1-discovery-runner.spec.ts` 入口；支持读取请求文件和初始化运行目录 | 第一段最小可启动骨架 | 能从请求文件启动第一段并生成空结果目录与过程文件 |
| Day 3 | 打通登录后探索能力 | 复用现有 Fixture 和登录能力，完成首页识别、菜单探索、页面截图、关键文案提取；控制探索范围只围绕目标场景 | 首页证据、菜单证据、页面截图、初步探索日志 | 能稳定拿到首页和目标页的基础证据，不再停留在纯自然语言层 |
| Day 4 | 生成结构化探索结果 | 把页面探索结果整理为字段候选、按钮候选、弹窗候选、表格列候选、成功提示候选；沉淀 `stage1-result.partial.json` | 第一段结构化中间结果 | 能从“页面截图/文本”进入“结构化候选结果” |
| Day 5 | 生成第二段任务草稿 | 开发 `task-draft-generator.ts`，将第一段结构化结果映射成 `draft.acceptance-task.json`；对不确定项打标，不直接写死为确定值 | 第一份第二段任务草稿 | 能生成可读、可改、可进入人工复核的任务 JSON 草稿 |
| Day 6 | 补齐人工复核包和持久化 | 输出 `review-notes.md`、缺失项说明、人工确认建议；复用现有持久化表记录第一段运行、快照、附件和草稿文件路径 | 复核包、第一段写库结果 | 第一段运行过程和产物可追踪，可复盘，可查询 |
| Day 7 | 联调第一段到第二段交接 | 选用现有“小区新增并回查”场景，执行“第一段生成草稿 -> 人工微调 -> 第二段执行”联调；修复首轮交接问题；补齐 README 和任务记录 | 首轮联调记录、问题清单、修订后的草稿样例 | 第一段至少能为一个真实场景生成可被第二段消费的任务文件 |

## 每日输出要求

每天结束后建议固定输出以下内容：

1. 当日代码改动说明
2. 当日新增文件列表
3. 当日未解决问题
4. 次日开发入口

建议每天下班前同步更新：

* `README.md`
* `.plans/`
* `.tasks/`

## 每日风险检查项

每天开发时都要复查以下问题：

* 是否引入了不必要的新目录或新依赖
* 是否破坏了第二段既有入口和运行习惯
* 是否把“不确定字段”误写成“确定配置”
* 是否输出了足够的人工复核信息
* 是否保留了完整运行证据和错误堆栈

## 本轮开发完成定义

本轮按天推进完成后，应满足以下结果：

* 第一段可以独立启动
* 第一段可以输出结构化证据
* 第一段可以生成第二段任务草稿
* 第一段有人工复核包
* 第一段结果可以进入第二段继续联调
