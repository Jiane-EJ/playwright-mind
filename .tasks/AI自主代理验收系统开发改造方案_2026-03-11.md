# AI 自主代理验收系统开发改造方案

## 基本信息

* 日期：2026-03-11
* 作者：Jiane
* 适用范围：当前 `HI-TEST` 仓库，先完成执行链路与任务编排，不涉及前端管理页面开发

## 1. 背景与目标

当前仓库已经具备 `Playwright + Midscene.js` 的基础运行能力，但仍停留在示例脚本阶段，尚未形成可复用的“任务输入 -> 场景执行 -> 结果输出”闭环。

本阶段目标不是做平台前端，而是先把下面几件事做扎实：

* 定义统一的测试任务输入格式
* 将自然语言业务场景拆分为可执行步骤
* 建立 Midscene + Playwright 的稳定执行框架
* 输出可追踪的执行日志、截图、HTML 报告和结构化结果
* 为后续前端页面、任务调度、批量回放预留标准接口

## 3. Midscene 官方能力边界与落地结论

参考官方文档：

* 简介：https://midscenejs.com/zh/introduction.html
* 在 Playwright 中使用：https://midscenejs.com/zh/integrate-with-playwright.html
* 使用 JavaScript 优化 AI 自动化代码：https://midscenejs.com/zh/usage/ai-native-automation-script.html
* 缓存与稳定性：https://midscenejs.com/zh/cache.html
* 常见问题：https://midscenejs.com/zh/faq.html
* 模型配置：https://midscenejs.com/zh/model-provider.html

结合官方文档，当前项目建议采用以下原则：

### 3.1 不要把整条业务流程写成一条超长 `ai()`

推荐按步骤拆成：

* `aiTap`
* `aiInput`
* `aiKeyboardPress`
* `aiScroll`
* `aiWaitFor`
* `aiQuery`
* `aiAssert`

原因：

* Vue 动态页面中菜单、弹窗、分页表格、异步刷新很多
* 一条长 Prompt 出错后难定位问题
* 步骤拆开后便于重试、截图、日志落盘和失败补偿

### 3.2 关键断言不要只依赖 `aiAssert`

官方明确建议，为降低 AI 幻觉风险，关键结果优先使用：

* `aiQuery` 提取结构化信息
* Playwright/TypeScript 代码做硬断言
* `aiAssert` 作为补充性可读断言

在“小区新增并回查”场景中，建议：

* 用 `aiQuery` 提取列表行数据
* 用代码断言新建的小区名称、状态、关键字段存在
* `aiAssert` 只作为页面层补充核验

### 3.3 动态页面稳定性要靠“结构化动作 + 等待策略 + 兜底”

建议组合：

* 页面跳转后用 `aiWaitFor` 等待首页或列表关键文本出现
* 菜单展开、弹窗出现、列表刷新后都增加显式等待
* 对高频定位目标增加 `cacheable` 策略，但不把缓存当唯一方案
* 对关键按钮保留候选描述词，例如“新增小区 / 新增 / 添加”

### 3.4 报告体系可以直接复用 Playwright + Midscene

建议保留现有：

* Playwright HTML 报告
* Midscene 报告目录
* 截图、日志、trace

后续再新增：

* `result.json`
* `step-results.json`
* `task-meta.json`

便于前端页面直接读取。

## 4. 推荐实现架构

本阶段按“无前端页面”的后端式执行架构推进。

### 4.1 总体链路

1. 人工填写任务 JSON
2. 任务校验器校验必填项、数据类型、步骤合法性
3. 执行器读取任务 JSON
4. 执行器按步骤调用 Midscene + Playwright
5. 每一步输出日志、截图、错误信息、提取结果
6. 汇总执行结果并生成结构化结果文件
7. Playwright/Midscene 报告用于人工复盘

### 4.2 模块拆分建议

建议新增如下目录：

```text
config/
  runtime-path.ts
  task-schema.ts
  env.ts

specs/
  tasks/
    acceptance-task.template.json
    acceptance-task.community-create.example.json

src/
  core/
    task-loader.ts
    task-validator.ts
    execution-context.ts
    result-writer.ts
  executor/
    scenario-runner.ts
    step-runner.ts
    action-runner.ts
    assert-runner.ts
  prompts/
    action-prompts.ts
    assert-prompts.ts
    query-prompts.ts
  utils/
    value-resolver.ts
    unique-data.ts
    mask.ts
  types/
    acceptance-task.ts
    execution-result.ts
tests/
  fixture/
    fixture.ts
  generated/
    acceptance-runner.spec.ts
```

### 4.3 核心职责说明

* `task-loader.ts`
  * 读取 JSON 文件
  * 解析环境变量占位符
  * 注入动态值，例如时间戳、随机后缀
* `task-validator.ts`
  * 校验 URL、账号信息、步骤类型、断言配置
  * 阻止无效任务直接进入执行阶段
* `scenario-runner.ts`
  * 管理整条任务生命周期
  * 负责前置校验、步骤编排、异常中断、结果汇总
* `step-runner.ts`
  * 逐步执行 `goto`、`login`、`navigateMenu`、`openDialog`、`fillForm`、`submit`、`searchTable`
* `result-writer.ts`
  * 输出结构化执行结果
  * 统一记录步骤耗时、状态、错误堆栈、截图路径

## 5. 任务输入模型设计

### 5.1 设计原则

* 测试输入先采用 JSON 文件，不依赖数据库
* 场景描述和测试数据分离
* 凭证信息允许临时直填，但生产建议走环境变量引用
* 表单字段必须结构化，不直接塞一整段自然语言
* 每一步都允许补充 `hints`，增强 Midscene 对页面元素的理解

### 5.2 建议字段

* `taskId`：任务唯一编码
* `taskName`：任务名称
* `target.url`：目标系统地址
* `account`：登录凭证
* `navigation.menuPath`：菜单路径
* `form.fields`：表单字段定义
* `assertions`：断言列表
* `cleanup`：失败或成功后的补偿动作
* `runtime`：浏览器、超时、截图等执行策略

### 5.3 为什么要结构化表单字段

如果只给一段自然语言，例如“录入完整后提交”，后续会遇到两个问题：

* 无法知道每个字段的值来源
* 无法稳定回查和失败补偿

所以建议把表单数据拆成：

* 字段标签
* 控件类型
* 值
* 是否必填
* 是否使用唯一值策略

## 6. 针对你当前示例场景的执行方案

你的示例场景可以拆成以下标准步骤：

1. 打开登录页
2. 输入账号密码并登录
3. 等待首页关键标识出现
4. 展开“小区管理”菜单
5. 点击“小区信息管理”子菜单
6. 等待小区管理页面加载完成
7. 点击“新增小区”
8. 等待新增弹窗出现
9. 按字段定义逐项录入表单
10. 点击“提交”
11. 校验提交成功提示
12. 关闭弹窗
13. 刷新或重新查询列表
14. 搜索刚新增的小区
15. 提取列表数据并做硬断言

这里最关键的是第 9、11、15 步：

* 第 9 步需要你提供完整字段清单
* 第 11 步需要明确成功提示词
* 第 15 步需要明确列表字段名称和搜索入口

## 8. 改造风险与问题清单

### 8.1 Midscene 版本与配置方式需要统一

当前仓库依赖：

* `@midscene/web`：`^0.9.2`

需要确认：

* 是否继续沿用当前版本
* 是否升级到官方最新稳定版本
* 升级后模型环境变量是否需要同步调整

如果版本不统一，后面会出现 API 差异、文档不一致、脚本示例不可直接复用的问题。

### 8.2 登录链路可能存在验证码、单点登录或二次确认

如果你的测试环境存在以下因素，必须提前确认：

* 图形验证码
* 短信验证码
* 企业微信 / CAS / SSO 登录跳转
* 首次登录修改密码

这些因素会直接决定登录步骤能否自动化。

### 8.3 新增小区表单字段目前不完整

目前你只给了流程，没有给字段明细，例如：

* 小区名称
* 所属区域
* 地址
* 联系人
* 联系电话
* 状态
* 经纬度
* 下拉框项

如果字段不全，执行器无法稳定填报，也无法设计唯一值回查。

### 8.4 动态表格回查要避免脆弱断言

高风险写法：

* 只断言“页面看起来新增成功了”
* 只断言 toast 出现

推荐写法：

* 用关键字搜索刚创建的小区名称
* 提取表格首行数据
* 对名称、状态等字段做代码断言

### 8.5 敏感数据不要长期固化在 JSON

联调阶段可以临时放账号密码，正式化时建议改为：

* JSON 中写 `${TEST_USERNAME}`、`${TEST_PASSWORD}`
* `.env` 或 CI Secret 注入实际值

## 12. 结论

这个方向可以做，而且当前仓库适合作为第一阶段基座，但必须先把“示例脚本工程”升级为“任务驱动执行工程”。

最重要的改造点只有四个：

* 输入结构化
* 步骤原子化
* 断言硬化
* 结果可沉淀

在不做前端页面的前提下，先用 JSON 驱动单场景联调，是当前性价比最高、风险最低的推进方式。

## 13. 双阶段统一落地（2026-03-11 补充）

根据最新确认，项目统一收敛为“同仓双阶段”模式，第一段与第二段既可串联，也可独立重复执行。

### 13.1 统一架构

* 第一段（探索建模）：
  * 输入自然语言和基础信息
  * AI 探索页面、截图、提取元素与流程
  * 人工补充与复核
  * 产出执行计划与任务 JSON
* 第二段（执行验证）：
  * 读取任务 JSON
  * Midscene + Playwright 执行步骤
  * 生成报告、日志、结构化结果

### 13.2 人工门禁机制

* 第一段探索完成后，不自动进入第二段
* 第二段默认可独立执行
* 当 `STAGE2_REQUIRE_APPROVAL=true` 时，任务 JSON 需声明：
  * `approval.approved=true`
  * 否则第二段直接拒绝执行

### 13.3 最小落地原则

* 先完成第二段最小执行器并稳定联调
* 第一段后续只需要产出标准任务 JSON 即可接入第二段
* 避免先开发前端页面和复杂调度，优先保证执行稳定性和可复盘能力
