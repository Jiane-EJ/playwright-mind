# 第二段数据持久化改造方案（2026-03-12）

## 背景

全局数据持久化底座已经创建完成，但当前第二段执行器仍然只输出文件结果，数据库尚未承载实际运行数据。

若不先接入第二段，后续页面与第一段设计都只能围绕静态 schema 展开，缺少真实运行数据样本。

## 目标

1. 保持当前第二段执行入口、文件产物和运行习惯不变。
2. 将第二段运行主记录、步骤明细、结构化快照和附件路径同步写入数据库。
3. 为后续页面、任务查询和第一段衔接提供真实数据底座。

## 写库范围

* `ai_task`：登记任务主信息
* `ai_task_version`：登记任务 JSON 版本
* `ai_run`：登记本次 Stage2 运行
* `ai_run_step`：记录每一步执行结果
* `ai_snapshot`：保存 `resolvedValues`、`querySnapshots`、进度状态
* `ai_artifact`：保存任务 JSON、`result.partial.json`、`result.json`、步骤截图路径
* `ai_audit_log`：记录运行开始、运行结束、步骤失败

## 实施方式

1. 新增 Stage2 专用持久化服务。
2. 在 `runTaskScenario` 中只接入三个节点：
   * 运行启动
   * 每步结束
   * 最终结果落盘
3. 所有写库动作采用最小侵入方式，不修改现有业务步骤和断言逻辑。
4. 若持久化初始化失败，只记录错误日志，不阻断现有 Stage2 执行。

## 约束

* 当前只支持 sqlite 写库
* 运行时自动补齐 migration，避免必须手工先执行初始化命令
* 当前不强依赖 Playwright HTML 报告与 Midscene 报告入库，只先记录当前运行目录下的核心产物
