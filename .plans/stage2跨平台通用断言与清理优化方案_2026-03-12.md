# Stage2 跨平台通用断言与清理优化方案（2026-03-12）

## 背景

当前 Stage2 已支持 JSON 驱动执行，但在多平台接入场景下，存在以下共性风险：

1. 断言部分存在 AI 语义误判空间，稳定性受页面波动影响较大。
2. 清理流程存在行匹配过宽导致误删的风险。
3. 平台间 DOM 差异（Element/Ant/其他）导致定位策略不可复用。

## 目标

1. 将断言能力统一为“Playwright 硬检测优先 + AI 结构化兜底”。
2. 将清理能力统一为“精确匹配 + 删除后校验”的安全模式。
3. 通过任务 JSON 的 `uiProfile` 提供跨平台选择器扩展点，避免代码硬编码平台差异。

## 方案

1. 在 `src/stage2/task-runner.ts` 增加通用配置解析：
   * `tableRowSelectors`
   * `toastSelectors`
   * `dialogSelectors`
2. 新增行匹配模式：
   * `exact`（默认，建议）
   * `contains`（仅在业务明确允许时使用）
3. 断言优化：
   * `table-row-exists` 增加空值防御与匹配模式。
   * `table-cell-equals` 强制基于结构化结果执行“列值比对”，避免仅凭 `matchedRow` 通过。
   * `table-cell-contains` 改为用返回值做代码级包含判断。
4. 清理优化：
   * 清理目标去重与空值过滤。
   * 删除时按 `rowMatchMode` 定位行。
   * 新增 `verifyAfterCleanup`，删除后校验目标行是否消失。
5. 模板与文档同步：
   * 更新 `specs/tasks/acceptance-task.template.json`
   * 更新示例任务 `specs/tasks/acceptance-task.community-create.example.json`
   * 更新 `README.md`

## 补充收敛项

1. `table-cell-equals` / `table-cell-contains` 的最终验收口径补充为：
   * 优先通过 Playwright 提取可见表格列值并做代码级比对
   * 仅在硬检测无法稳定完成时再降级到 AI 结构化断言
2. `form.successText` 对应的“检查提交提示”步骤必须基于真实检测结果决定通过/跳过，不能出现未检测到仍记为 `passed` 的情况。
3. 最终验收口径补充为：
   * `table-row-exists` 作为硬门槛，优先验证“数据确实已生成并可查询”
   * `table-cell-equals` / `table-cell-contains` 只建议校验少量关键列
   * 明细列校验默认建议配置 `soft=true`，避免因列表列渲染差异阻断整条任务

## 回滚

1. 任务侧回滚：
   * `cleanup.verifyAfterCleanup=false`
   * `cleanup.rowMatchMode=contains`（不建议，仅临时）
2. 代码侧回滚：
   * 回退 `task-runner.ts` 断言与清理改动。
