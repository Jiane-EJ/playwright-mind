# 全局数据持久化方案（2026-03-12）

## 背景

当前项目已经具备第二段执行能力，但所有核心结果仍主要依赖文件目录落盘，后续若增加页面展示、任务查询、执行追踪和跨阶段关联，单纯扫描 `t_runtime/` 会逐步失控。

因此需要先建立统一的数据持久化底座，作为后续第二段写库、第一段建模、页面查询和 MySQL 迁移的公共基础。

## 目标

1. 在项目内创建本地数据库底座，优先满足本地单文件运行。
2. 表结构和字段设计兼容后续迁移到 MySQL。
3. 文件系统继续保留运行产物，数据库负责结构化数据与索引。
4. 第一阶段与第二阶段后续统一基于同一套持久化模型扩展。

## 选型

* 本地数据库：`SQLite`
* 运行方式：Node `node:sqlite`
* 迁移原则：
  * 主键统一使用应用侧生成的字符串 ID
  * 时间字段统一使用 `DATETIME`
  * JSON 快照统一先存 `TEXT`
  * 不使用 SQLite 专属表特性和 SQL 方言

## 数据边界

数据库中保存：

* 任务主信息
* 任务版本与 JSON 内容
* 阶段运行主记录
* 步骤执行明细
* 结构化快照
* 附件元数据与路径
* 审计日志

文件系统中保留：

* 截图
* HTML 报告
* Midscene 报告
* `result.json`
* `result.partial.json`
* 其他运行附件

## 核心表

* `ai_task`
* `ai_task_version`
* `ai_run`
* `ai_run_step`
* `ai_snapshot`
* `ai_artifact`
* `ai_audit_log`

## 实施顺序

1. 先创建数据库配置与 migration 基线
2. 再改造第二段执行器，将现有运行结果双写入库
3. 最后基于该底座设计第一段完整模型

## 约束

* 当前只落本地 `SQLite` 驱动，不在本轮直接支持 MySQL 连接
* 但 schema 必须从第一版开始避免未来 MySQL 迁移返工
* 第二段接库时不能破坏现有文件产物输出
