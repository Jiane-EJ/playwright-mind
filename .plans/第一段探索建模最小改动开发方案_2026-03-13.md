# 第一段探索建模最小改动开发方案

## 基本信息

* 日期：2026-03-13
* 作者：Jiane
* 适用范围：当前 `HI-TEST` 仓库第一段“探索建模”能力补齐，目标是最小改动接入现有第二段执行体系

## 1. 背景

当前仓库已经完成第二段主链路，包括：

* `Stage2` JSON 任务执行入口
* 运行产物目录收敛
* 本地 `SQLite` 持久化底座
* 第二段运行、步骤、快照、附件写库

但“同仓双阶段”中的第一段仍未落地。现阶段如果直接开发前端页面、任务中心或复杂调度，会明显扩大改动范围，也会绕开当前已经稳定下来的第二段能力。

因此第一段必须按“最小改动”原则推进：

* 不先做前端页面
* 不先做复杂调度
* 不破坏当前第二段入口和数据结构
* 优先复用现有 `Playwright + Midscene + runtime-path + persistence`

## 2. 第一段目标

第一段不是替代第二段，而是为第二段稳定产出“可人工复核的任务草稿”。

本轮第一段的最小目标只有四个：

1. 输入自然语言场景和基础信息
2. 自动探索目标页面并沉淀证据
3. 自动生成第二段任务 JSON 草稿
4. 产出人工复核包，人工确认后再交给第二段执行

## 3. 最小改动原则

### 3.1 复用现有能力

第一段应直接复用以下已有能力：

* `tests/fixture/fixture.ts` 中的 Midscene + Playwright Fixture
* `config/runtime-path.ts` 的运行目录拼装方式
* `db/migrations/001_global_persistence_init.sql` 的通用持久化表
* `src/persistence/sqlite-runtime.ts` 的数据库和 migration 能力
* 第二段已有的任务文件路径、运行目录、附件归档思路

### 3.2 本轮不做

本轮第一段明确不做：

* 前端配置页面
* 页面化人工审核台
* 多任务并发调度
* MySQL 真实接入
* 自动直接下发第二段执行
* 大规模结构重构

### 3.3 本轮允许新增

为落地第一段，允许新增但必须控制在最小范围内：

* `src/stage1/` 目录
* 第一段输入模板
* 第一段运行入口
* 第一段结构化输出文件
* 第一段写库适配层

## 4. 第一段完整流程

第一段完整流程建议固定为以下九步：

1. 读取第一段请求文件，加载场景描述、目标地址、账号、范围约束和人工审批信息
2. 执行基础校验，确认目标地址、凭证、入口说明、建模目标是否齐全
3. 启动浏览器并进入目标系统，完成登录及首页识别
4. 基于自然语言目标执行“受控探索”，只探索当前场景需要的菜单、表单、表格、按钮和提示信息
5. 采集探索证据，包括页面截图、关键文案、弹窗标题、搜索区字段、表格列名、操作按钮、成功提示候选文案
6. 将证据转换为结构化中间结果，形成“步骤草稿 + 字段草稿 + 断言草稿 + 清理草稿”
7. 基于中间结果生成第二段任务 JSON 草稿，不直接覆盖正式任务文件
8. 输出人工复核包，由人工补全不确定项并确认是否可进入第二段
9. 人工确认后，将最终任务 JSON 交给第二段执行入口继续联调

## 5. 第一段输入与输出

### 5.1 输入

为保持最小改动，第一段输入先继续采用文件方式，而不是页面录入。建议新增：

* `specs/stage1/stage1-request.template.json`
* `specs/stage1/stage1-request.community-create.example.json`

建议输入字段：

* `requestId`
* `requestName`
* `target.url`
* `account.username`
* `account.password`
* `goal.scenarioDescription`
* `goal.expectedOutcome`
* `scope.menuPathHints`
* `scope.mustExploreAreas`
* `scope.skipAreas`
* `review.required`
* `review.reviewer`

### 5.2 输出

第一段运行后，建议统一输出到独立目录，例如：

* `t_runtime/stage1-results/<requestId>/<timestamp>/`

该目录中至少包含：

* `stage1-result.json`
* `stage1-result.partial.json`
* `evidence/` 页面截图与关键附件
* `draft.acceptance-task.json` 第二段任务草稿
* `review-notes.md` 人工复核说明

说明：

* `draft.acceptance-task.json` 只作为草稿，不直接替换 `specs/tasks/` 中的正式文件
* 人工确认后，再将草稿复制或整理为第二段正式任务文件

## 6. 模块拆分建议

在尽量不动现有第二段的前提下，第一段建议新增以下最小模块：

```text
src/
  stage1/
    types.ts
    request-loader.ts
    explorer.ts
    evidence-writer.ts
    task-draft-generator.ts
    review-package-writer.ts
    stage1-runner.ts

tests/
  generated/
    stage1-discovery-runner.spec.ts
```

各模块职责建议如下：

* `types.ts`：定义第一段请求模型、探索结果模型、草稿任务模型
* `request-loader.ts`：读取第一段请求文件并做最小字段校验
* `explorer.ts`：封装登录后探索、页面截图、文案提取、元素候选提取
* `evidence-writer.ts`：统一落盘中间证据与过程快照
* `task-draft-generator.ts`：将第一段结构化结果转成第二段任务 JSON 草稿
* `review-package-writer.ts`：输出人工复核说明、缺失项清单和不确定项说明
* `stage1-runner.ts`：编排第一段全生命周期

## 7. 与现有持久化的复用方式

第一段建议直接复用现有通用持久化表，不单独改库：

* `ai_task`：登记第一段来源任务
* `ai_task_version`：保存第一段输入和生成草稿的版本快照
* `ai_run`：记录 `stage_code=stage1`
* `ai_run_step`：记录探索步骤
* `ai_snapshot`：记录元素候选、字段候选、步骤草稿、中间状态
* `ai_artifact`：记录截图、草稿 JSON、人工复核说明路径
* `ai_audit_log`：记录开始、完成、人工确认等关键事件

最小改动要求下，不新增新表，优先通过：

* `stage_code`
* `artifact_type`
* `snapshot_key`

区分第一段与第二段。

## 8. 第一段到第二段的交接规则

### 8.1 交接前提

第一段产物进入第二段前，必须满足：

* 已生成 `draft.acceptance-task.json`
* 已输出 `review-notes.md`
* 不确定项已被标注
* 人工确认通过

### 8.2 交接方式

本轮不做自动串行执行，只做“人工门禁后再交接”：

1. 第一段生成草稿
2. 人工补全或修正
3. 人工将最终文件放入 `specs/tasks/`
4. 第二段按现有入口执行

这样做的原因是：

* 不改当前第二段执行入口
* 不引入自动串联后的额外失败链路
* 先保证第一段建模结果可读、可审、可修

## 9. 第一段实施顺序

第一段建议按以下顺序开发：

1. 定义输入模型和输出产物格式
2. 搭建第一段最小执行入口
3. 打通登录后探索和证据采集
4. 输出结构化中间结果
5. 生成第二段任务 JSON 草稿
6. 输出人工复核包
7. 接入持久化和运行快照
8. 用现有“小区新增并回查”场景做首轮联调

## 10. 风险与控制

### 10.1 风险

第一段主要风险有：

* AI 探索结果不稳定，导致生成草稿字段不全
* 页面元素命名与业务术语不一致，导致字段映射错误
* 探索范围过大，导致运行时长过长、证据噪音过多
* 自动生成的任务草稿看似完整，但实际缺少可执行细节

### 10.2 控制策略

控制策略建议如下：

* 第一版只支持单场景、单页面链路
* 第一版必须带人工复核包，不允许“探索完直接执行”
* 第一版优先围绕现有示例场景联调
* 第一版字段生成优先保守，不确定就标注“待人工确认”

## 11. 完成标准

第一段本轮完成标准建议定义为：

1. 能读取第一段请求文件并正常启动
2. 能完成目标页面受控探索并生成截图和结构化证据
3. 能自动生成一份第二段任务 JSON 草稿
4. 能输出人工复核说明，明确哪些字段已确认、哪些字段待补充
5. 能将第一段运行信息写入现有持久化底座
6. 能通过人工修订后的任务文件进入第二段继续执行
